# プロジェクト総括報告書スキル

プロジェクト完了後の総括報告書を、インタビュー形式でコンテンツを収集し、プロフェッショナルなDOCXとして生成する。

## ワークフロー概要

```
Phase 1: 設定確認 → Phase 2: コンテンツ収集 → Phase 3: 構成・レビュー → Phase 4: DOCX生成
```

---

## Phase 1: 設定確認

以下の3項目をユーザーに確認する。まとめて質問してよい。

### 1-1. 読者タイプ

| タイプ | 対象 | 特徴 |
|--------|------|------|
| **社内向け** | 上司・経営層・チーム内共有 | 数値を具体的に記載。課題も率直に。社内用語OK。コスト・工数の透明性 |
| **顧客向け** | クライアント・納品先 | 成果・価値を前面に。技術詳細は適度に抽象化。課題は「対処済み」として前向きに表現。専門用語には解説を付ける |

読者タイプは報告書全体のトーン・詳細度・表現に影響する。

### 1-2. セクション選択

以下から含めるセクションを選択してもらう（番号指定、「全部」「1,2,3,7,11」等）。
カスタムセクションの追加も可。

| # | セクション | 内容 |
|---|-----------|------|
| 1 | プロジェクト概要 | 名称・期間・背景・目的・成果サマリー |
| 2 | 技術構成 | 言語・FW・DB・インフラの技術スタック |
| 3 | アーキテクチャ | 構成図・設計方針・データフロー |
| 4 | 開発プロセス | 手法・フロー・CI/CD・品質管理 |
| 5 | ユースケース | 主要な利用シナリオ・ユーザーストーリー |
| 6 | 成果物 | 納品物一覧・品質指標 |
| 7 | 課題・教訓 | 問題と対処・得られた教訓 |
| 8 | 改善履歴 | 改善活動の時系列記録 |
| 9 | 導入効果 | ROI・KPI・Before/After |
| 10 | 競合との比較 | 差別化ポイント・比較マトリクス |
| 11 | 今後の展望 | 次期計画・ロードマップ・推奨事項 |

**推奨**: セクション1（プロジェクト概要）は常に含めることを推奨する。
**順序**: デフォルトは上記番号順。ユーザーの希望に応じて並べ替え可能。

### 1-3. スタイル選択

| スタイル | 特徴 |
|---------|------|
| **シンプル** | 白黒基調。最小装飾のクリーンなビジネス文書 |
| **ブランドカラー** | ユーザー指定の primary/accent カラーを適用。ロゴ挿入可 |
| **フォーマル** | ネイビー系アクセント。格式ある保守的レイアウト |
| **モダン** | 鮮やかアクセント。ゆったり余白の現代的デザイン |
| **スタイリッシュ** | 太字・高コントラスト。ビビッドアクセントの現代的デザイン |
| **エグゼクティブ** | チャコール x ゴールド。経営層向けの洗練されたデザイン |
| **テック** | ダークテーマ風。エンジニアリング報告向け |

「ブランドカラー」選択時は追加で以下を確認:
- プライマリカラー（16進数、例: 1B4F72）
- アクセントカラー（16進数、例: E74C3C）
- ロゴ画像パス（任意）

### 設定確認

3項目の回答後、設定サマリーを表示してユーザーに確認する。

```
【設定サマリー】
- 読者: 社内向け
- セクション: 1, 2, 3, 7, 11
- スタイル: フォーマル
```

## Phase 2: コンテンツ収集

`references/section-templates.md` を読み込み、選択された各セクションについてヒアリングを行う。

### ヒアリングの進め方

1. 選択セクションを順番にヒアリングする
2. 各セクションについて `section-templates.md` に定義された質問（3-5問）を提示する
3. 読者タイプ（社内/顧客）に応じた質問を選ぶ
4. ユーザーの回答形式は自由: テキスト、ファイルパス指定、URL、「あとで埋める」等
5. 各セクションの回答後、収集内容を要約して確認

### 自動検出の提案

プロジェクトディレクトリの情報から自動取込を提案する:

| 対象 | 検出条件 | 該当セクション |
|------|---------|--------------|
| `improvement_list/*.md` | ディレクトリが存在 | 改善履歴 (8) |
| `package.json` / `requirements.txt` 等 | ファイルが存在 | 技術構成 (2) |
| `README.md` | ファイルが存在 | プロジェクト概要 (1) |
| `docker-compose.yml` | ファイルが存在 | アーキテクチャ (3) |
| `.github/workflows/` | ディレクトリが存在 | 開発プロセス (4) |

自動検出はあくまで提案であり、ユーザーの承認を得てから取り込む。

### コンテンツ深度の調整

| 観点 | 社内向け | 顧客向け |
|------|---------|---------|
| 数値 | 具体的（コスト、工数、遅延日数） | 効果中心（削減率、ROI） |
| 課題 | 率直に記載。原因分析含む | 「対処した課題」として前向きに |
| 技術 | 詳細（バージョン、設定値） | 概要レベル（目的と効果中心） |
| 体制 | 個人名・役割を明記 | 役割ベースで記述 |

## Phase 3: 構成・レビュー

### Markdownドラフトの作成

1. 収集したコンテンツをMarkdown形式で報告書全体のドラフトを作成
2. 構成:
   - 表紙情報（タイトル、サブタイトル、日付、作成者/組織名）
   - 目次（「※DOCX生成時に自動生成」と注記）
   - 選択された各セクション
   - 必要に応じて付録（詳細データ、改善履歴一覧等）
3. ドラフトをユーザーに提示

### Mermaid図の埋め込み

アーキテクチャ（セクション3）、開発プロセス（セクション4）、ユースケース（セクション5）が選択されている場合、適切なMermaid図をコードブロックとして生成する。

- ` ```mermaid ` フェンスで記述し、直前にキャプション行（例: `**図3-1: システム構成図**`）を付ける
- 対応する図の種類: `graph/flowchart`, `sequenceDiagram`, `C4Context`, `gitGraph`, `gantt`
- 図の候補は `references/section-templates.md` の各セクション「Mermaid図の候補」を参照
- ユーザーはレビュー時に図の修正・削除が可能

### レビューと修正

- ユーザーのフィードバックに基づいて修正
- 「この部分を詳しく」「ここは削除」「トーンを変えて」等に対応
- ユーザーが承認するまで繰り返す
- 承認後、Phase 4 へ進む

---

## Phase 4: DOCX生成

### 前提

DOCX生成のメカニクス（docx-jsのセットアップ、スタイル定義、テーブル、リスト、画像、ヘッダー/フッター、目次、バリデーション）は **docxスキル**（`C:/work/utility/skills-main/skills-main/skills/docx/SKILL.md`）のパターンに従うこと。以下は報告書固有のレイアウト指示のみ記載する。

### スタイル値の適用

`references/style-presets.md` を読み込み、選択されたスタイルの値を使用する。

### ドキュメント構造

```
[表紙]
  - プロジェクト名（タイトルサイズ、中央揃え）
  - サブタイトル「プロジェクト総括報告書」
  - 日付
  - 作成者/組織名
  - [ブランドカラー] ロゴ画像（右上）
  --- 改ページ ---

[目次]
  - TableOfContents（Heading1-3）
  --- 改ページ ---

[本文]
  - 各セクション: Heading1 で開始
  - サブセクション: Heading2, Heading3
  - 表、箇条書き、段落を適切に配置

[付録]（必要な場合のみ）
  - Heading1「付録」で開始
```

### ページ設定

- 用紙: A4（11906 x 16838 DXA）
- マージン: 上下左右 1440 DXA（1インチ）
- ヘッダー: プロジェクト名（右寄せ、本文より小さいフォント）
- フッター: ページ番号（中央揃え）

### Mermaid図のレンダリングと埋め込み

Markdownドラフト内の Mermaid コードブロックを画像に変換し、DOCXに埋め込む。

#### 前提

- `@mermaid-js/mermaid-cli` (mmdc) がグローバルインストールされていること
- 未インストールの場合: `npm install -g @mermaid-js/mermaid-cli`

#### 変換手順

1. ドラフト内の各 ` ```mermaid ` ブロックを一時 `.mmd` ファイルとして保存
2. スタイルプリセットに応じた mmdc テーマ設定ファイル (`mermaid-config.json`) を生成（`references/style-presets.md` の「Mermaid図のスタイル連動」参照）
3. mmdc でPNGに変換:

```bash
mmdc -i diagram.mmd -o diagram.png -w 1600 -b transparent --scale 2 -c mermaid-config.json
```

4. 生成されたPNGを `ImageRun` で埋め込み、直後にキャプション用 `Paragraph`（captionサイズ、accentカラー、中央揃え）を配置
5. 一時ファイル (.mmd, .png, mermaid-config.json) はDOCX生成後に削除

#### 図のサイズガイドライン

- 全幅図: width 500pt（コンテンツ幅に収まる範囲）
- ハーフ幅図: width 250pt
- アスペクト比は mmdc 出力の実寸から計算

#### エラーハンドリング

- mmdc 未インストール: ユーザーにインストールを案内し、図なしで続行するか確認
- レンダリング失敗: Mermaid構文エラーを修正して再試行。3回失敗したらテキストベースの説明に代替
- PNG未生成: 該当図をスキップし続行

### docxスキルのクリティカルルール遵守

以下を必ず守ること（docxスキルより）:
- `\n` を使わず、別の `Paragraph` 要素を使用
- テーブルは `WidthType.DXA` を使用（`PERCENTAGE` は不可）
- テーブル幅はテーブルレベル + セルレベルの両方で指定
- `ShadingType.CLEAR` を使用（`SOLID` は不可）
- 見出しスタイルに `outlineLevel` を含める（目次生成に必須）
- Unicode箇条書き記号を使わず `LevelFormat.BULLET` を使用
- `ImageRun` には `type` パラメータを指定

### バリデーション

生成後、以下で検証:
```bash
python C:/work/utility/skills-main/skills-main/skills/docx/scripts/office/validate.py <output.docx>
```

### 出力

- ファイル名のデフォルト: `{プロジェクト名}_総括報告書_{YYYYMMDD}.docx`
- 出力先はユーザーに確認（デフォルト: カレントディレクトリ）
