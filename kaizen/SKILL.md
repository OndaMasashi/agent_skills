# Kaizen: 継続的改善

**原則**: 「多くの小さな改善は、1つの大きな変更に勝る。エラーは修正ではなく設計段階で防ぐ。」

## 4つの柱

### 1. 継続的改善 (Kaizen)

- 最小の実行可能な変更から始める
- 一度に1つの改善
- 各変更を検証してから次へ
- コードは見つけた時より良くして去る
- 反復的洗練: 動かす → 明確にする → 効率化する（同時にやらない）

### 2. ポカヨケ (Error Proofing)

エラーを実行時ではなくコンパイル/設計時に防ぐ:

- **型システム**で制約（コンパイル時）
- **バリデーション**をシステム境界で即座に（実行時、早期）
- **ガード**で前提条件を明示
- **エラー境界**で優雅な劣化
- Result<T, E>を返し、throwを避ける
- 不正な状態を型で表現不可能にする

### 3. 標準化 (Standardized Work)

- 既存のコードベースパターンに従う
- 解決済みの問題を再発明しない
- 新パターンは大幅に優れている場合のみ
- リンター/型チェック/テスト/CI/CDで自動的に品質を担保

### 4. JIT (Just-In-Time)

- **YAGNI**: 今の要件だけを実装する
- 「念のため」のコードは書かない
- プロファイルしてから最適化する
- 3つ以上の類似ケースが出るまで抽象化しない（Rule of Three）

## レッドフラグ

| 柱 | 違反の兆候 |
|-----|-----------|
| 改善 | 「後でリファクタリングする」（実行されない） |
| ポカヨケ | 「ユーザーが気をつければいい」 |
| 標準化 | 既存パターンを確認せずに独自実装 |
| JIT | 「いつか必要になるかもしれない」 |

## 適用場面

- コード実装・リファクタリング
- アーキテクチャ・設計判断
- 業務プロセス改善提案（中小企業コンサル）
- エラーハンドリング・バリデーション設計
