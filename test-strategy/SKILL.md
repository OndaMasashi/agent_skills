# テスト戦略ガイド（考慮観点集）

このスキルは **テスト戦略を検討する際の観点集** であり、全項目を実装する指示ではない。YAGNI を踏まえ、PJ の規模・性質に応じて採用可否を判断する。

## 使い方

1. **判断フロー**（§0）でまず PJ の位置づけを確認し、必須／推奨／任意の範囲を決める
2. ピラミッド（§1）・静的解析（§2）は原則必須
3. §3〜§5 はスタックに応じて観点を抽出
4. §6 は補強領域。PJ 要件と照らして採否を決定
5. 最後にチェックリスト（§まとめ）で網羅性を確認

## 0. 判断フロー

```
PJ 規模 / 種別
├─ 個人PoC・1日使い捨て     → §1 ピラミッド把握のみ（テスト省略可）
├─ 小規模（数百LOC・短命）   → §1 + §2 静的解析 + §3 ユニットテストのみ
├─ 中規模（本番投入）        → 上記 + §3〜§5 の主要パターン + §7 CI
└─ 大規模・ミッションクリティカル → 上記 + §6 から該当する補強領域
```

## 1. テストピラミッドと各層の役割

ソフトウェアテストは層構造。各層にコスト・速度・信頼性のトレードオフがある。

| 層 | 対象範囲 | 速度 | コスト | 得られる信頼性 |
|---|---|---|---|---|
| 静的解析 | 全ソースファイル | 高速 (s) | 低 | 型安全性・スタイル統一 |
| ユニットテスト | 単一関数・クラス | 非常に高速 (ms) | 低 | ロジックの正確性 |
| 統合テスト | 複数モジュール + DB | 高速 (s) | 中 | コンポーネント間連携 |
| E2E テスト | ブラウザ + サーバー全体 | 低速 (10s+) | 高 | 実ユーザーフロー |

**基本原則**: ユニット多め、統合ほどほど、E2E 最小限。静的解析はコストゼロなので全ファイルに適用。

## 2. 静的解析・型チェック

テストでは見つけにくいバグ（型不一致・未使用変数・missing await 等）を検出。テストより先に導入すべき最初の品質基盤。

### 2.1 基本構成

任意の言語で以下 3 要素を揃える:

- **Linter**: コード品質・典型的な落とし穴を検出
- **Formatter**: コードスタイル統一
- **Type checker**: 型整合性を静的検証

### 2.2 ツール例: Python

- Ruff (lint + format を 1 ツールで提供、推奨初期ルール: `E,W,F,I,UP,B,SIM`)
- mypy（段階的に厳格化: 最初は `disallow_untyped_defs=false` で start → 徐々に厳しく）

### 2.3 ツール例: TypeScript

- ESLint（ロジックルール）+ Prettier（フォーマット）+ tsc（型）
- eslint-config-prettier を ESLint 設定末尾に配置してフォーマット競合を無効化

### 2.4 pre-commit フック

コミット前にチェックを自動実行。壊れたコードがリポジトリに入るのを防ぐ。

### 2.5 CI パイプライン

push/PR 時に全チェックを自動実行。バックエンドとフロントエンドは並列ジョブ化。`concurrency` グループで連続 push 時の重複実行をキャンセル。

## 3. バックエンドテスト

### 3.1 共通 Fixture パターン

- **インメモリ DB**: ファイル I/O なしにテストごとにクリーンな DB を提供（例: `sqlite:///:memory:`）
- **モック Repository / 外部 API クライアント**: 外部依存を排除し、事前設定したデータを返す

### 3.2 ユニットテスト

純粋関数・バリデーション・計算ロジックを単独でテスト。DB・ネットワーク・副作用不要。

**原則**: 決定的・高速・単一責任。テスト名は期待する動作を説明。

### 3.3 統合テスト

複数コンポーネントが DB を含めて連携するかを検証。

### 3.4 非同期テスト

非同期ランタイム上のコードには専用フィクスチャ/モードを使用（例: pytest-asyncio の `asyncio_mode=auto`）。

### 3.5 並行性テスト

共有リソースが並行アクセス下で正しくロックされるかを検証。race condition シナリオ・冪等性ガードをテスト。

### 3.6 ロギング・監査テスト

構造化ログ（structlog 等）の `capture_logs()` で監査イベントの出力を検証。

---

## 4. フロントエンドテスト

### 4.1 状態管理ストアのテスト

コンポーネントをレンダリングせず、ストアの setState / getState を直接呼んでロジックを検証。
beforeEach でストア状態をリセットしテスト間の分離を確保。

### 4.2 API クライアントテスト

グローバル fetch 関数をスタブし、ネットワークなしで API 層を検証。期待 URL・クエリ・ヘッダをアサート。

### 4.3 コンポーネントテスト

レンダリング結果から表示内容をアサート。操作は user-event。
**原則**: 実装ではなく動作をテスト。CSS クラス・test-id ではなく **role / text / label** でクエリ。

### 4.4 API 契約テスト

フロントが送信するデータ形式がバックエンドの期待と一致することを検証。
**効果**: 単位のミスマッチ（例: 外部単位 vs 内部単位、ms vs s）は純粋なユニットテストでは見逃す。

---

## 5. E2E テスト

### 5.1 設定

- 共有状態を持つテストは `workers: 1` でシリアル実行
- `trace: on-first-retry` / `screenshot: only-on-failure` で失敗時のみ診断情報
- `baseURL` と timeout を明示

### 5.2 ヘルパー / ページオブジェクト

再利用アクションをヘルパー関数に抽出し spec 間の重複を削減。

### 5.3 ステートフルフローのシリアル実行

セットアップ → 操作 → クリーンアップ の依存がある場合は `mode: "serial"` を使用。

## 6. 補強領域（PJ 要件に応じて採否判断）

| 領域 | 得られるもの | 適用すべき場面 |
|---|---|---|
| 6.1 コードカバレッジ計測 | 未テスト領域の定量化 | 中規模以上。**下限値として使い目標値にしない** |
| 6.2 セキュリティスキャン | 依存/コードの脆弱性検出 | 本番投入・外部公開 PJ は必須 |
| 6.3 API スキーマバリデーション | F/B 乖離の検出 | API 公開・複数クライアントを持つ PJ |
| 6.4 パフォーマンス/ベンチマーク | クリティカルパスの性能保証 | レスポンス SLA がある PJ |
| 6.5 ビジュアルリグレッション | 意図しない UI 変更の検出 | ブランドサイト・頻繁な UI 改修 PJ |
| 6.6 ミューテーションテスト | テストの抜け穴検出 | ビジネスクリティカルなコアロジックのみ |
| 6.7 負荷/ストレステスト | ピーク負荷下の挙動保証 | 高トラフィック・SLA がある PJ |

**注意**: カバレッジは必要条件であって十分条件ではない。100% でも正確性は保証されない。

## 7. CI パイプライン設計

### 推奨ジョブ構成

```
push/PR -> [backend ジョブ]  -> lint -> type check -> test
        -> [frontend ジョブ] -> lint -> format check -> type check -> test
        -> [e2e ジョブ] (任意、両方通過後)
```

### キャッシュ戦略

依存関係ロックファイルのハッシュをキーに、ダウンロードキャッシュを保存。

### 同時実行制御

同一ブランチへの新コミット時、実行中 CI をキャンセル:
```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```

## まとめ: テストチェックリスト

PJ のテスト成熟度評価に使用。

| カテゴリ | 必須 | 推奨 | あると望ましい |
|---|---|---|---|
| 静的解析 | リンター + フォーマッター | 型チェッカー | セキュリティスキャン |
| ユニットテスト | 純粋関数・計算ロジック | エラーパス・エッジケース | プロパティベーステスト |
| 統合テスト | DB + サービス層 | マルチサービスフロー | 外部 API モック |
| E2E テスト | 主要ユーザーフロー | エラーシナリオ | ビジュアルリグレッション |
| CI パイプライン | PR で lint + テスト | 型チェック + カバレッジ | パフォーマンスゲート |
| pre-commit | lint + format | 型チェック | コミットメッセージ lint |
