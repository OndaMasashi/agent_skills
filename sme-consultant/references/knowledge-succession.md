## SECIモデル（知識創造）

### 概要
野中郁次郎・竹内弘高らが提唱した知識創造のフレームワーク。暗黙知と形式知を循環させ、組織の知識を蓄積・発展させる。

### 4つのプロセス

| プロセス | 変換 | 内容 |
|---------|------|------|
| 共同化（Socialization） | 暗黙知→暗黙知 | OJT、シャドーイング、同行・同席で経験を共有 |
| 表出化（Externalization） | 暗黙知→形式知 | マニュアル化、チェックリスト化、動画・図解への落とし込み |
| 連結化（Combination） | 形式知→形式知 | 複数の形式知を統合、標準プロセス・ナレッジベース構築 |
| 内面化（Internalization） | 形式知→暗黙知 | 研修・実践を通じて「体で覚える」状態に |

### ナレッジ承継での活用ポイント

- **共同化＋表出化**でベテランの暗黙知を形式知に変換するのがスタート
- **連結化**で体系化・検索性向上・標準化を進める
- **内面化**で次世代の暗黙知を再生成し、スパイラルを回し続ける

### 現場での具体イメージ

- 共同化：熟練作業者と新人のペア作業、営業同行、クレーム対応の同席
- 表出化：作業動画＋コメント、営業トーク台本化、判断基準のチェックリスト化
- 連結化：社内Wiki分類・タグ付け、業務フローとナレッジの紐付け
- 内面化：ロールプレイ、現場演習、振り返りミーティング

## DRINKフレームワーク（IT化設計）

### 概要
ナレッジ承継をIT化する際の設計観点を整理するチェックリスト的フレームワーク。

### 5つの要素

| 要素 | 設計観点 |
|------|---------|
| D: Database | どこに・どう蓄積するか（ナレッジDB、構造化、検索性） |
| R: Real-time/Report | どう更新し、どう見える化するか（リアルタイム収集、ダッシュボード） |
| I: Interface | 現場が使いたくなるUI/UXか（複数の入口、業務フロー連動） |
| N: Network | 誰と誰をどうつなぐか（エキスパート情報、コミュニティ） |
| K: Knowledge | 何をナレッジとして扱うか（対象範囲、テンプレート定義） |

### SECIモデルとの組み合わせ

- 「どのプロセスの成果物をどのDBに入れるか」→ D
- 「誰がどこからアクセスするか」→ I・N
- 「どの頻度で更新・レビューするか」→ R
- 暗黙知を形式知にする「場」の仕組み化 → K

### 製造現場での適用例

- D：動画＋テキストの作業手順をSharePoint/Notion等に構造化して蓄積
- R：新人からの質問ログ・不良発生レポートを自動収集、FAQに反映
- I：現場タブレットから工程別・トラブル別にすぐ引けるUI
- N：各ラインのリーダー・熟練者を「問い合わせ先」として登録
- K：標準手順と熟練者Tipsの区分を定義、テンプレート統一

## 技術承継へのAI活用

### 全体設計：RAG＋ファインチューニングの二段構え

| 手法 | 特徴 | 向いているケース |
|------|------|-----------------|
| RAG | ナレッジをDB化しLLMで検索。データ追加で拡張容易 | 手順・事例が頻繁に変わる、大量ナレッジへのアクセス性向上 |
| ファインチューニング | ドメイン用語・判断基準をモデルに染み込ませる | 基本ルールは安定、判断基準・語彙の均質化 |

### RAGの具体活用案

1. **ナレッジポータル／QAボット**
   - 作業手順書、トラブル事例集、ベテランインタビュー記録をベクトル化
   - 「この症状のとき何を確認すべきか」に過去記録から回答

2. **作業動画＋暗黙知の構造化**
   - ベテラン作業を動画撮影、音声をテキスト化・要約
   - 「音が高くなったら圧上がりすぎ」等の口伝えコツをナレッジ化

3. **トラブルシューティング支援**
   - 過去の故障・不具合事例DBから類似事例と対処手順を提示

### ファインチューニングの具体活用案

1. **自社流の判断ロジック埋め込み**
   - 合否判定、危険度ランク付け等の教師データ作成
   - 「ベテランならこうコメントする」出力で判断一貫性向上

2. **専門用語・文書スタイル標準化**
   - 内部用語・略称・言い回しを学習し報告書テンプレ生成に活用

### PoCステップ案

1. ユースケース選定（最も質問が多いテーマ1〜2個）
2. 既存データの整理（マニュアル・日報・トラブル記録のテキスト化）
3. RAGミニマム版構築（部署限定でクローズド運用）
4. 利用ログ収集（質問・回答・ベテラン訂正）
5. ピンポイントでファインチューニング（小規模LoRA等）

## 事業承継・PMI

### 事業承継の基本ポイント

| 観点 | 内容 |
|------|------|
| 目的と将来像 | 維持重視か成長重視か、5〜10年後の姿を言語化 |
| 経営資源の棚卸し | 人・資産・知的資産（理念、技術、顧客関係、ブランド、許認可） |
| 後継者育成 | 早期決定、理念・暗黙知の時間をかけた伝承 |
| 見える化・磨き上げ | 財務・事業・組織の現状を開示可能レベルに整理 |
| ステークホルダー説明 | 従業員・取引先・金融機関への目的・方針の事前共有 |

### PMI（Post Merger Integration）の設計ポイント

- **プレPMI**から統合方針・100日プランを用意
- **統合範囲**を明確化（経営、業務、IT・システム、人事・制度）
- **成長型か持続型か**の方針選択
- **ガバナンス**と意思決定プロセスの再設計
- 中小PMIガイドライン・専門家の活用

### PMI現場での重点テーマ

| テーマ | 内容 |
|--------|------|
| 組織文化・価値観 | 働き方・ルール・評価軸の違いを言語化、共通ルール策定 |
| 人事・人材統合 | 役割・処遇・評価の整理、キープレイヤーの流出防止 |
| 業務プロセス再設計 | 重複業務の洗い出し、標準化・統合フロー設計 |
| IT・システム統合 | 基幹システムの統合方針と移行手順 |
| 信頼関係構築 | 経営者による直接説明、不安・疑問を出せる場づくり |

### フェーズ別の要点

| フェーズ | 内容 |
|----------|------|
| 準備（プレPMI） | 目的・方針共有、課題仮説整理、統合計画作成 |
| 初期〜100日 | 新方針提示、キーマンとの対話、クイックウィン創出 |
| 1年以内 | 業務・システム統合、組織再編、KPIで成果見える化 |
| 1〜数年 | 文化定着、追加成長投資、次世代経営チーム育成 |

### 中小企業ならではの着眼点

- オーナー経営者の「想い」を形式知化（理念ストーリー、意思決定基準）
- 取引先集中・キーパーソン集中リスクへの対処
- 人材・時間リソースの制約に配慮し、優先順位を3〜5項目に絞る
